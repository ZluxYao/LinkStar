1. 统一添加服务商信息，这样子就可以直接选

2. 可以指定多久更新一次 （或者有更好的方案）

3. 当stun模块发现本地ip变化，关联的ip就更新

4. 可以自定义从哪里获取ip，stun 某个网站，和某个域名保持一致

流程

## 一、核心抽象

整个模块围绕一个动作：**拿到当前 IP → 和记录的 IP 比对 → 变了就调服务商 API 更新**。
差异只在两点：IP 从哪来、什么时候触发。归并成两个抽象：

### 1. IPSource（对应需求 4：自定义 IP 来源）
每条 DDNS 记录绑定一个「IP 来源」，统一接口 `Resolve(recordType) (ip, err)`：
- `stun`：直接读 `stun.Runtime.Network.PublicIP/LocalIP`，零成本，不自己探测
- `web`：HTTP GET 某网站（如 https://ip.sb）
- `dns`：解析某域名，保持一致
- `interface`：读本地网卡
- A 记录取 IPv4，AAAA 取 IPv6

### 2. Trigger（对应需求 2、3：触发时机）
定时和事件只是同一个 `syncRecord()` 的两个触发器，不写两套逻辑：
- **事件触发（优先）**：STUN 检测到本地 IP 变化 → 发 hook → DDNS 订阅 → 立即同步。
  仿 `stun/hooks.go` 模式加 `EmitNetworkChanged`，stun 不感知 ddns，避免循环 import。
  触发点在 `stun/network.go:83` 已判断 IP 变化处。
- **定时触发（兜底）**：全局 ticker，防止 hook 漏掉、或 web/dns 来源变了但 STUN 没动。

## 二、调度模型（最终方案）

### ticker：单 ticker + due-time 扫描
- **只有一个全局 ticker，10s 扫一次**（扫描粒度，不是同步间隔）。
  不要每条记录 / 每个服务商一个 ticker —— 那会导致 goroutine 泄漏、增删记录要起停 ticker、并发写 config。
- 每条记录记「上次检查时间」，扫描时判断 `now - LastCheckAt >= interval` 才同步。
- 自定义间隔 ≠ 多实例：间隔差异靠 due 判断实现，记录在哪个间隔无所谓，永远一个 ticker。
- **时间被量化成 10s 的格子**：设 293s 和 296s 都落在 300s 这一格，同一批更新。
  实际生效间隔 = `ceil(interval/10)*10`。
- 间隔字段：全局默认 300s（`DDNSConfig.IntervalSec`）+ 记录级可选覆盖（`r.IntervalSec`，0 表示用全局）。
  一次覆盖「固定全局」和「每条自定义」两种需求，以后不返工。

### 遍历：遍历记录，不遍历服务商
- 调度最小单位是「一条记录的同步」，服务商只是记录的属性（`ProviderID`），用到时查表。
- Records 是扁平一维 slice，直接扫最简单。

### 服务商实例：一个服务商共享一个实例（两阶段）
- `Cloudflare` 是无状态的（只持有 token + httpClient），httpClient 内部有连接池要复用。
- **阶段一（启动）**：遍历 `Config.Providers` 建表 `map[ProviderID]DNSProvider`，
  每个 ProviderID 建一个实例，一对一天然不重复。
- **阶段二（运行）**：scheduler 遍历 Records，`providerClients[r.ProviderID]` 只查不建。
- 实例唯一性由 map 的 key 保证，与遍历顺序、记录位置无关。
- 多条记录指向同一 ProviderID → 共享同一指针（同一 httpClient、token、未来的 zone 缓存）。
- 用户增删改 Provider 时重建这张表（`UpdateConfig` 后调 `rebuildProviderClients()`）。

### 并发：服务商间并发 + 服务商内串行
- Cloudflare 按账号（token）限流（1200 req/5min）。同账号并发会撞同一限流桶，不同账号互不相关。
- 数据结构天然对应：**按 ProviderID 分组，组间并行，组内串行**。
- 每个 provider 实例自带一个 worker goroutine + channel，启动时 `go w.run(ctx)`。
  N 个服务商 = N 个常驻 goroutine，数量固定可控，不泄漏。
- scheduler 扫描只负责「投递」到对应 worker 的 channel，不直接同步。
  channel 满（该服务商在忙）就 `default` 跳过，下轮 due 判断会自然重投，不阻塞 scheduler。
- 启动首次同步和运行时扫描走同一条路（scan→投递），不写两套。

## 三、状态与失败处理

### 状态存储：只存一条最新值，不存历史
- `Last*` 字段（LastIP / LastStatus / LastMessage / LastSyncAt / LastCheckAt）每次覆盖，不追加、不存数组。
- 理由：config 要整体 WriteJsonFile 写回，历史数组会无限膨胀、拖慢写盘。
- **config 存状态（现在怎样），logrus 存历史（过去发生过什么）**。
- 失败原因用 `logrus.Warnf` 落日志文件，config 只留最新 `LastMessage`。

### 失败重试：靠 ticker 自然轮询，区分临时/永久
不写独立重试 goroutine，10s ticker 本身就是重试器，只控制「失败后下次要不要再被扫到」：
- **临时性**（网络超时、5xx、IP 来源解析失败）：不更新 LastIP，下个 due 周期自动重试。
- **永久性**（token 错、域名不存在、401/403）：标记 failed 并暂停（`Enabled=false` 或 `Paused`），
  等用户改配置，不再自动重试。
- `LastCheckAt` 无论成败都更新（代表「检查过了」），`LastSyncAt` 只在真正调 API 成功后更新。
- 防瞬时抖动的「连续两次一致才认」先不做（STUN 自己已做梯度检查），等真遇到再加。

## 四、字段与文件落点

### model（ddns_config.go）补充字段
- `DDNSConfig.IntervalSec`：全局默认间隔，300
- `DDNSRecord.IPSource` / `IPSourceArg`：IP 来源类型 + 参数（web 的 URL / dns 的域名）
- `DDNSRecord.IntervalSec`：记录级覆盖，0 用全局
- `DDNSRecord.LastCheckAt`：用于 due 判断（已有 LastSyncAt 用于展示）
